### 数据流向（一次测试请求的完整旅程）

```
[Newman] 读取 Collection 中的第 N 条请求
    │
    ├─ 1. 执行 pre-request 脚本（如：自动往请求头塞 token）
    │
    ├─ 2. 替换 {{变量}}（如 {{base_url}} → http://localhost:3000）
    │
    ├─ 3. 发送 HTTP 请求 ───────────────► [Mock Server:3000]
    │                                          │
    │                                          ├─ 检查 token 是否有效
    │                                          ├─ 查 db.json 数据库
    │                                          ├─ 执行业务逻辑
    │                                          └─ 返回 JSON 响应
    │    ◄─────────────────────────────────────┘
    │
    ├─ 4. 收到响应，执行 Tests 脚本（断言）
    │      pm.test("状态码是 200", ...)
    │      pm.test("返回了 token", ...)
    │
    └─ 5. 记录结果：通过 ✓ 或 失败 ✗
```

## 二、按制作顺序，每一步做了什么、为什么

### 阶段 0：准备工作

#### 第 1 步：创建项目文件夹 `OrderAPITestProject`

```
你做了什么：右键 → 新建文件夹 → 命名为 OrderAPITestProject
```

**技术上发生了什么：** 操作系统在硬盘上分配了一块空间，用一个"路径"（`d:\OrderAPITestProject`）来指向它。以后所有文件都会放在这个路径下。

**为什么要做：** 把项目的所有文件放在一起，方便管理。就像你不会把作业本、铅笔、橡皮扔在不同的房间。

#### 第 2 步：安装 Node.js

```
你做了什么：去 nodejs.org 下载安装包，双击安装
```

**技术上发生了什么：**
- 你的电脑上多了一个叫 `node.exe` 的程序
- 同时还自带了一个叫 `npm` 的工具（Node Package Manager，包管理器）
- `npm` 可以一键下载别人写好的代码库（叫"包"或 package）

**为什么要做：** 我们后面用的 Mock Server（json-server）、Newman 都是 Node.js 生态里的工具，没有 Node.js 就跑不了。

**类比：** Node.js 像是给你的电脑安装了一个"万能播放器"，npm 像是应用商店，可以一键安装各种"插件"。

```
你做了什么：根据 API 规格，把每个接口的请求格式、响应格式、错误码整理成文档
```

#### 第 4 步：编写 `docs/测试计划.md`

```
你做了什么：制定测试策略——要测哪些东西、怎么测、谁测、什么时候测完
```

**为什么要做：** 需求分析是"这个接口长什么样"，测试计划是"我要怎么去验证它"。就像你拿到菜单后，要决定先尝哪个菜、怎么判断好不好吃。

#### 第 5 步：编写 `docs/测试用例.md`

```
你做了什么：写了 42 条详细的测试用例，每条包含
  - 输入什么数据
  - 期望返回什么状态码和内容
  - 用"等价类划分"和"边界值分析"的方法设计
```

**等价类划分（用人话说）：**
把无穷多的可能输入，分成几个"有代表性的组"，每组挑一个代表来测。
比如"密码长度必须是 6-64 位"：
- 有效等价类：6-64 位 → 挑 10 位来测
- 无效等价类 1：< 6 位 → 挑 5 位来测
- 无效等价类 2：> 64 位 → 挑 65 位来测

**边界值分析（用人话说）：**
bug 最喜欢藏在边界上。所以要测：
- 最小值本身（6 位）
- 最小值 -1（5 位）
- 最大值本身（64 位）
- 最大值 +1（65 位）

**`npm init -y` 做了什么：** 在当前文件夹生成了一个叫 `package.json` 的文件。这个文件记录了：

#### 第 7 步：安装 json-server

```bash
npm install json-server
```

**json-server 是什么：** 一个神器——给它一个 JSON 文件，它就能在 30 秒内生成一个完整的 REST API 服务器。你不需要手写数据库、不需要手写增删改查代码。

**为什么用它：** 真实的后端可能还没开发好，或者测试环境不稳定。Mock Server 就是"替身演员"——长相、动作都和真人一样，但完全由你控制。你可以随时重置数据、模拟各种异常。

**`npm install` 做了什么：**
1. 去 npm 的中央仓库（registry.npmjs.org）下载 json-server 的代码
2. 放到 `node_modules/` 文件夹里
3. 在 `package.json` 里记录"我依赖了 json-server"
4. 生成 `package-lock.json`，锁定精确版本（防止以后版本变了出问题）

#### 第 8 步：编写 `db.json`——模拟数据库

```json
{
  "users": [
    { "id": 1, "username": "testuser", "password": "Test@123456", "token": "..." }
  ],
  "orders": [
    { "id": 10001, "status": "PENDING", ... }
  ]
}
```

**db.json 是什么：** 就是"假数据库"。json-server 会把这个文件当作数据库来读写。你往里面加数据，API 就能查到；你修改它，API 返回的内容就会变。

**为什么要精心设计数据：**
- 需要一个 PENDING 订单 → 用来测"取消订单"功能
- 需要一个 SHIPPED 订单 → 用来测"已发货不能取消"
- 需要一个 CANCELLED 订单 → 用来测"已取消不能重复取消"
- 需要一个属于其他用户的订单 → 用来测"越权访问"

每种状态都像"道具"，测试就是"演戏"——你没准备好道具，戏就演不下去。

#### 第 9 步：编写 `server.js`——自定义服务器逻辑

```javascript
const jsonServer = require('json-server');
const server = jsonServer.create();
const router = jsonServer.router('db.json');
// ...
server.listen(3000, () => {
  console.log('Mock Server 已启动');
});
```

**每一行都在干什么（逐个解释）：**

| 代码 | 人话翻译 |
|------|---------|
| `require('json-server')` | "把我之前 npm install 的 json-server 拿过来用" |
| `jsonServer.create()` | "创建一个空的服务器壳子" |
| `jsonServer.router('db.json')` | "告诉服务器：你的数据库就是这个 db.json 文件" |
| `server.use(...)` | "给服务器装上中间件（过滤器）" |
| `server.post('/login', ...)` | "当有人往 /login 这个地址发 POST 请求时，执行这段代码" |
| `req.body.username` | "从请求体里取出 username 字段" |
| `res.json({...})` | "把这段 JSON 数据当作响应发回去" |
| `res.status(401)` | "把响应状态码设为 401（未授权）" |
| `server.listen(3000, ...)` | "服务器开始工作，守在 3000 号端口，有人敲门就响应" |

**req 和 res 是什么：**

```
req = request = 请求 = 客户端发来的消息
res = response = 响应 = 你发回去的消息

客户端："你好，我要登录，用户名 testuser，密码 Test@123456"
         ──────────── req.body ──────────────
         
服务器："好的，这是你的 token：eyJhbG..."
         ──────────── res.json(...) ─────────
```

**req 里能取到什么：**
- `req.body` → POST/PUT 请求携带的 JSON 数据
- `req.params` → URL 里的动态参数，如 `/orders/:id` 中的 id
- `req.headers` → 请求头（里面有 Authorization token）
- `req.query` → URL 问号后面的参数，如 `?page=1`

**res 能做什么：**
- `res.json(obj)` → 返回 JSON 数据
- `res.status(n)` → 设置 HTTP 状态码
- `res.send(text)` → 返回纯文本

#### 第 9.5 步：编写认证守卫（Auth Guard）

```javascript
server.use((req, res, next) => {
  if (!req.path.startsWith('/orders')) return next();  // 不是订单接口，放行

  const auth = req.headers.authorization;
  if (!auth || !auth.startsWith('Bearer ')) {
    return res.status(401).json({ code: 2001, message: '未提供认证令牌' });
  }

  const token = auth.replace('Bearer ', '');
  const user = db.get('users').find({ token }).value();
  if (!user) {
    return res.status(401).json({ code: 2001, message: '未提供认证令牌' });
  }

  req.currentUser = user;  // 把当前用户信息挂在 req 上，后面的代码可以用
  next();  // 放行！
});
```

**这个"守卫"在做什么：** 就像门卫大爷：

1. 不是去订单区域的？→ 放行
2. 去订单区域但没有门禁卡（token）？→ 拦下，返回 401
3. 门禁卡过期了？→ 拦下，返回 401
4. 门禁卡是伪造的？→ 拦下，返回 401
5. 一切正常 → 放行，并在小本本上记下"你是谁"

**`next()` 是什么意思：** Express 的中间件是一层层过滤的。`next()` 就是"我这关过了，交给下一关"。不调 `next()` 就是"到我这儿就打住了，后面不用看了"。

### 阶段 3：Postman Collection

#### 第 11 步：生成 Postman Collection

```
你做了什么：用 postman-test-generator 技能生成了一个 .postman_collection.json 文件
```

**Postman Collection 的本质是什么：**

它是一个 **JSON 格式的测试剧本**。里面描述了：
- 46 个"请求"，每个请求包含：
  - URL（请求发到哪里）
  - Method（GET/POST/PUT/DELETE）
  - Headers（附加信息，如 token）
  - Body（携带的数据）
  - pre-request script（发送前执行的 JavaScript 代码）
  - test script（收到响应后执行的 JavaScript 断言代码）

你可以把它想象成"拍电影的分镜头脚本"：
- 第 1 场：登录，预期返回 200 和 token
- 第 2 场：创建订单，预期返回 201
- ...

**为什么 Newman 能执行它：** 因为 Postman Collection 的格式是公开的标准（v2.1.0 schema），就像一个"通用乐谱"。Postman 桌面版能演奏，Newman（命令行版）也能演奏——它们能看懂同一份乐谱。

**Collection 内部结构（简化版）：**
```json
{
  "info": { "name": "订单系统 API 测试" },
  "item": [
    {
      "name": "TC-LOGIN-001 合法登录",
      "event": [
        {
          "listen": "test",              // 这是"收到响应后执行"的脚本
          "script": {
            "exec": [
              "pm.test('状态码是200', function() {",
              "  pm.response.to.have.status(200);",
              "});"
            ]
          }
        }
      ],
      "request": {
        "method": "POST",
        "url": { "raw": "{{base_url}}/login" },
        "body": { "raw": "{\"username\":\"testuser\",\"password\":\"Test@123456\"}" }
      }
    }
  ]
}
```

每一个 `item` 就是一个测试场景，`pm.test(...)` 就是一条断言——"我预期结果是 X，如果不是，就报错"。

#### 第 12 步：编写 `dev.environment.json`

```json
{
  "name": "OrderAPI-Dev",
  "values": [
    { "key": "base_url", "value": "http://localhost:3000", "enabled": true },
    { "key": "username", "value": "testuser", "enabled": true },
    { "key": "password", "value": "Test@123456", "enabled": true },
    { "key": "auth_token", "value": "", "enabled": true },
    { "key": "order_id", "value": "", "enabled": true }
  ]
}
```

**环境变量有什么用：**

在 Collection 里，你不写死 `http://localhost:3000`，而是写 `{{base_url}}`。这样：
- 在开发环境，`base_url = http://localhost:3000`
- 在测试环境，`base_url = https://test-api.example.com`
- 在生产环境（只读），`base_url = https://api.example.com`

一个 Collection，配合不同的 environment 文件，可以测试不同的服务器。就像同一个剧本，换不同的舞台演出。

**`auth_token` 和 `order_id` 为什么初始为空：** 它们不是手动填的，而是由脚本自动设置的：
- 登录成功后，Tests 脚本里写 `pm.environment.set("auth_token", jsonData.data.token)`
- 创建订单成功后，Tests 脚本里写 `pm.environment.set("order_id", jsonData.data.id)`

这就是"自动化"的核心——数据在测试用例之间流转，不需要人工干预。

### 阶段 4：Newman 执行

#### 第 13 步：安装 Newman

```bash
npm install -g newman
```

- `-g` 表示 global（全局安装），装完后你可以在任何目录使用 `newman` 命令
- Newman 是 Postman 的命令行版本——没有图形界面，全靠命令

**Newman 和 Postman 桌面版的关系：**

| | Postman 桌面版 | Newman |
|---|---|---|
| 界面 | 漂亮的图形界面 | 纯命令行，黑底白字 |
| 用途 | 手动调试、编写请求 | 自动化批量执行 |
| 适用场景 | 开发阶段调试 | CI/CD 流水线 |
| 谁在操作 | 人点鼠标 | 机器自动跑 |
| 核心引擎 | 相同 | 相同 |

它们用的是同一套 JavaScript 运行环境（sandbox），所以你在 Postman 里写的 pre-request script 和 test script，Newman 都能正确执行。

### 阶段 5：自动化缺陷推送

#### 第 15 步：编写 `auto-create-bugs.js`

**这个脚本的工作流程：**

```
1. 读取 newman/report.json
       │
       ▼
2. 找到所有失败的断言
   执行结果结构：
   {
     "run": {
       "executions": [
         { "item": { "name": "TC-LOGIN-002" },
           "assertions": [
             { "assertion": "Status code is 200", "error": {...} },
             { "assertion": "code equals 0", "error": {...} }
           ]
         }
       ]
     }
   }
       │
       ▼
3. 按请求名称分组，每个失败的请求 → 一个 TAPD 缺陷
       │
       ▼
4. 构造缺陷内容（标题、描述、请求/响应详情）
       │
       ▼
5. 调用 TAPD API：POST https://api.tapd.cn/bugs
   请求头：Authorization: Basic <base64编码的账号:密码>
   请求体：{ workspace_id, title, description, severity, priority }
       │
       ▼
6. 打印结果：哪个缺陷创建成功，哪个失败
```

**为什么 JSON 报告能作为中间产物：**

JSON（JavaScript Object Notation）是一种"人也能读、机器也能读"的数据格式。它用 `{}` 表示对象，`[]` 表示数组，`"key": "value"` 表示键值对。

Newman 输出 JSON 报告 → `auto-create-bugs.js` 读取 JSON 报告 → 提取信息 → 调用 TAPD API。每一步各司其职，互不干扰。

JSON 就像一个"通用快递盒"——不管里面装什么，只要格式统一，谁都能拆开看懂。

#### 第 16 步：了解 TAPD API 调用

**REST API 是什么：**

REST 是一种设计风格——用 HTTP 方法（GET/POST/PUT/DELETE）来操作资源。

| HTTP 方法 | 对应的数据库操作 | 例子 |
|-----------|----------------|------|
| GET | 读（SELECT） | 查看缺陷详情 |
| POST | 增（INSERT） | 创建一个新缺陷 |
| PUT | 改（UPDATE） | 更新缺陷状态 |
| DELETE | 删（DELETE） | 删除缺陷 |

每个资源有一个唯一的 URL 地址：
```
GET    https://api.tapd.cn/bugs          → 获取缺陷列表
POST   https://api.tapd.cn/bugs          → 创建新缺陷
GET    https://api.tapd.cn/bugs/12345    → 查看 12345 号缺陷
PUT    https://api.tapd.cn/bugs/12345    → 修改 12345 号缺陷
```

**Basic Auth 怎么工作：**

```
客户端想访问 TAPD API，需要证明"我是谁"。

1. 客户端把 "api_user:api_password" 用 Base64 编码
   "your-name@company.com:ABCD-EFGH-IJKL-MNOP"
   编码后 → "eW91ci1uYW1lQGNvbXBhbnkuY29tOkFCQ0QtRUZHSC1JSktMLU1OT1A="

2. 在请求头中加上：
   Authorization: Basic eW91ci1uYW1lQGNvbXBhbnkuY29tOkFCQ0QtRUZHSC1JSktMLU1OT1A=

3. 服务器收到后，解码 → 验证账号密码 → 通过就处理请求
```

**注意：** Base64 不是加密！它只是"换一种写法"，任何人都可以解码回原文。所以 Basic Auth 必须配合 HTTPS（加密传输），否则账号密码就会在网络上裸奔。

**`workspace_id` 是什么：**

TAPD 里有很多项目空间（workspace），每个有独立的 ID。你在 URL 里看到的 `https://www.tapd.cn/12345678/...` 里的那串数字就是 workspace_id。API 需要知道"你要在哪个项目里创建缺陷"。

## 三、每个技术角色的深入解释

### 3.1 Postman Collection 到底是什么？

一个 JSON 文件，是"测试剧本"的标准化表达。包含：
- 请求定义（URL、方法、头、体）
- 前置脚本（pre-request：发请求前要做什么）
- 测试脚本（tests：收到响应后怎么判断对不对）
- 变量（{{base_url}} 这类占位符）

**为什么能被 Newman 执行：** Collection 遵循公开的 v2.1.0 格式规范。Newman 内置了一个解析器，能读取这个 JSON，然后逐条发送 HTTP 请求并执行脚本。Postman 桌面版和 Newman 用的是同一套运行时引擎。

---

### 3.2 Newman 是什么？

Newman = Postman 的命令行版本。没有 GUI，全靠终端。

- Postman 桌面版：适合写请求、调试、手动测试
- Newman：适合批量跑、定时跑、在服务器上跑、在 CI 里跑

你可以把 Newman 想象成"自动演奏钢琴"——你给它乐谱（Collection），它自动弹，不需要人坐在琴前面。

---

### 3.3 Mock 服务器深入

**"服务器"是什么：** 一台一直运行着的程序，等待别人来请求，然后给出响应。就像 24 小时便利店——你随时去，随时有人（程序）接待你。

**"监听端口"是什么意思：**

服务器运行在一台电脑上，这台电脑只有一个 IP 地址（像大楼地址）。但电脑上可能同时跑着多个服务器（Web 服务器、数据库、Mock Server）。怎么区分请求是给谁的呢？——靠端口号！

```
IP 地址 = 大楼地址
端口号   = 房间号

http://localhost:3000
        │          └── 端口号（房间 3000）
        └── localhost = 本机（127.0.0.1）
```

常见端口号：
- 80：HTTP 网站
- 443：HTTPS 网站
- 3306：MySQL 数据库
- 3000：开发服务器（约定俗成）

**`localhost` 是什么：** 就是"本机"，一个特殊的域名，指向 `127.0.0.1`——你的电脑自己。访问 `localhost:3000` 就是在自己电脑上找 3000 端口的服务。不需要联网。

**req/res 的工作流程（完整版）：**

```
1. 客户端（Newman）发送请求：
   POST http://localhost:3000/login
   Body: {"username":"testuser","password":"Test@123456"}

2. 网络层：数据被打包成 TCP 数据包，发到 127.0.0.1:3000

3. 服务器（Express）收到请求：
   - 解析 HTTP 报文 → 生成 req 对象
   - 匹配路由 → 找到 server.post('/login', ...) 对应的处理函数

4. 处理函数执行：
   const { username, password } = req.body;
   // username = "testuser", password = "Test@123456"

5. 查数据库：
   const user = db.get('users').find({ username, password }).value();

6. 构造响应：
   res.json({ code: 0, data: { token: "..." } });
   // Express 把 JSON 对象序列化为字符串，设置 Content-Type: application/json

7. 客户端收到响应，解析 JSON，执行断言
```

---

### 3.4 TAPD API 调用深入

**HTTP 请求的结构：**

一个 HTTP 请求包含三部分：
1. **起始行**：`POST /bugs HTTP/1.1`（方法 + 路径 + 协议版本）
2. **请求头**：键值对，描述元信息
   ```
   Host: api.tapd.cn
   Authorization: Basic eW91ci1uYW1l...
   Content-Type: application/json
   ```
3. **请求体**（可选）：实际的数据
   ```json
   {
     "workspace_id": 12345678,
     "title": "[AutoTest] TC-LOGIN-002 — 状态码错误",
     "severity": "normal"
   }
   ```

**HTTPS 的作用：** HTTP + SSL/TLS 加密。中间的任何人（路由器、WiFi 提供者、ISP）都看不到你在传什么内容。TAPD API 用的是 HTTPS，所以你的 api_password 在传输过程中是安全的。

---

### 3.5 批处理脚本 `.bat` 的串联原理

`.bat` 文件是 Windows 的批处理脚本。本质上就是一个文本文件，里面每行是一条命令，Windows 会从上到下逐行执行。

```bat
@echo off              REM 关闭命令回显
cd /d d:\OrderAPITest  REM 切换到项目目录
newman run ...         REM 执行第一件事
node auto-create-bugs  REM 执行第二件事
```

`&&` 和 `||` 的区别（在 Windows cmd 中不支持，用 `if errorlevel` 代替）：
- `cmd1 && cmd2`：cmd1 成功才执行 cmd2
- `cmd1 || cmd2`：cmd1 失败才执行 cmd2
- 在 PowerShell 中写法不同（用 `; if ($?) { ... }`）

## 四、5 个最容易混淆的关键概念（傻瓜式比喻）

### 概念 1：API 是什么？（餐厅比喻）

```
你去餐厅吃饭：
  1. 你看菜单（API 文档），上面写着"宫保鸡丁，辣度可选"
  2. 你告诉服务员"一份宫保鸡丁，微辣"（发送 API 请求）
  3. 服务员把订单传给厨房（服务器处理请求）
  4. 厨房做好菜，服务员端上来（服务器返回响应）
  5. 你尝了一口，"嗯，确实是微辣"（断言通过）

API 就是"服务员"——它是你和厨房之间的沟通桥梁。
你不用知道厨房里有什么锅、什么灶（后端实现细节），
只需要告诉服务员你要什么，服务员就会给你端来。
```

### 概念 2：端口是什么？（大楼门牌号比喻）

```
一台服务器（电脑）就像一个大型商场：
  - 商场地址 = IP 地址（如 192.168.1.100）
  - 每家店铺的门牌号 = 端口号

  :80   → 星巴克（Web 服务器）
  :3000 → 奶茶店（Mock 服务器）
  :3306 → 仓库（MySQL 数据库）
  :8080 → 服装店（另一个 Web 服务）

你去商场，光知道地址不够，还要知道去哪家店。
http://192.168.1.100:3000 = "去这个商场的 3000 号店铺"
```

### 概念 3：Token 是什么？（演唱会手环比喻）

```
你去演唱会：
  1. 门口验票，给你戴上一个手环（登录，拿到 token）
  2. 你出去上厕所，回来时保安看手环就放行（后续请求带 token）
  3. 不需要每次都掏身份证（不需要每次输密码）

Token 就是一个"临时凭证"，证明"我已经登录过了"。

为什么要有过期时间？
  手环只在今天有效——你不可能拿着去年的手环混进今年的演唱会。
  Token 过期了（code=2002），表示你需要重新登录。

Bearer 是什么意思？
  "Bearer" 就是"持有者"——谁拿着这个 token，谁就是它的主人。
  就像手环——谁戴着谁就能进场，不管手环最初是给谁的。
```

### 概念 4：JSON 是什么？（快递单比喻）

```json
{
  "寄件人": "测试系统",
  "收件人": "Mock Server",
  "包裹内容": {
    "username": "testuser",
    "password": "Test@123456"
  }
}
```

```
XML 时代（旧格式）：就像一张写满格式的表格，有很多重复的标签
  <user>
    <username>testuser</username>
    <password>Test@123456</password>
  </user>

JSON（新格式）：就像便条，简洁直白
  { "username": "testuser", "password": "Test@123456" }

JSON 为什么成了主流？
  1. 比 XML 短很多（省流量、省时间）
  2. 和 JavaScript 的对象语法一模一样（前端直接能用）
  3. 人类看着也很直观
```

### 概念 5：同步 vs 异步（去银行办事比喻）

```
同步（阻塞）：你在柜台办业务，柜员说"等一下，我去后台查"，
  然后你就站在柜台前干等，后面所有人也都等着。
  → 这段时间什么都做不了，程序"卡住"了。

异步（非阻塞）：你取号后坐在椅子上等，轮到你了广播叫你。
  在等待期间你可以刷手机、看报纸。
  → 程序不会卡住，可以同时处理其他事情。

JavaScript 的 pm.sendRequest 就是异步的：
  pm.sendRequest({...}, function(err, res) {
    // 这个 function 是"回调函数"——等请求完成后才执行
    pm.environment.set("order_id", res.json().data.id);
  });
  // 程序不会停在这里等，而是继续往下走
  // 请求完成后，自动调用回调函数处理结果
```

**为什么测试脚本里用异步：**

```
TC-CANCEL-002 的 pre-request 要做的事：
  1. 创建一个新订单（POST /orders）        ← 异步，不阻塞
  2. 把新订单的 ID 存到环境变量
  3. 然后 Newman 发送真正的请求（PUT /orders/:id/cancel）

如果 1 是同步的（阻塞的），整个测试就会卡在创建订单这一步。
```

**回调函数是什么：** "你办完了叫我一声，我告诉你接下来做什么"。你把电话号码留给柜员（注册回调函数），柜员办完了给你打电话（调用回调函数）。

## 五、Bug 修复全记录：每个报错的原因和修复逻辑

这部分解释上次修复中每一个失败的原因——为什么失败、怎么修、为什么这么修。

### 修复 1：TC-LOGIN-002 登录失败（username=4 字符边界）

**现象：** 用 `abcd` + `Test@123456` 登录，期望 200，实际返回 401。

**原因：** db.json 里 `abcd` 用户的密码是 `T@1234`，不是 `Test@123456`。测试脚本混用了 `{{password}}` 变量（值为 `Test@123456`）和边界用户名 `abcd`，它们不是一对。

**修复：** 把请求体中的 `"password": "{{password}}"` 改为 `"password": "T@1234"`。

**为什么这么改：** 测试的目的是验证"4 字符用户名能正常登录"，不是验证特定用户。只要存在一个 4 字符用户并且用正确的密码，测试就应该通过。

---

### 修复 2：TC-LOGIN-004 登录失败（password=6 字符边界）

**现象：** 用 `testuser` + `T@1234` 登录，期望 200，实际 401。

**原因：** 和上面相反——`testuser` 的密码是 `Test@123456`，不是 `T@1234`。

**修复：** 把请求体中的 `"username": "{{username}}"` 改为 `"username": "abcd"`，让 username 和 password 配对。

---

### 修复 3：TC-LOGIN-005 登录失败（password=64 字符边界）

**现象：** 用 `testuser` + 64 字符密码登录，期望 200，实际 401。

**原因：** 没有一个用户的密码是这 64 个字符。这是一个"孤儿测试用例"——测试数据在 db.json 中没有对应的用户。

**修复：** 
1. 在 `db.json` 里新增用户 `pwdtest`，密码就是那 64 个字符
2. 把测试请求中的 username 从 `{{username}}` 改为 `pwdtest`

**为什么不能直接改 testuser 的密码：** 因为 testuser 是主测试用户，被其他几十个测试用例依赖。改了它的密码会导致大量其他测试失败。

---

### 修复 4：TC-ORDER-002 — remark 字段默认值错误

**现象：** 创建订单时不传 remark，期望 `remark` 为 null，实际返回 `""`（空字符串）。

**原因：** Mock Server 中这一行：
```javascript
remark: remark || ''
```
JavaScript 中 `undefined || ''` 返回 `''`。`||` 运算符会返回第一个"真值"，而 `undefined` 是"假值"，所以返回了空字符串。

**修复：** 改为：
```javascript
remark: remark || null
```

**null 和空字符串有什么区别：**
- `null` 表示"故意不填，没有值"
- `""` 表示"填了，但填的是空内容"
- `undefined` 表示"根本没提供这个字段"

API 设计中，可选字段不传时应该返回 null，这样前端能区分"用户没填"和"用户填了空"。

---

### 修复 5：TC-GET-004 — 不存在的订单返回了 200

**现象：** 查询 `GET /orders/99999999`，期望 404 + code=3001，实际返回 200 + `{"code":0,"data":{}}`。

**原因：** json-server 对不存在的资源返回 `{}`（空对象），然后被 `router.render` 包装成了成功响应。在 JavaScript 中：
```javascript
!!{}          // true —— 空对象是 truthy！
!![]          // true —— 空数组也是 truthy！
!!""          // false
!!0           // false
!!null        // false
!!undefined   // false
```

所以原来 `if (!data)` 判断能过滤掉 `null` 和 `undefined`，但过滤不掉 `{}`。

**修复：**
```javascript
const isEmpty = !data
  || (Array.isArray(data) && data.length === 0)         // 空数组
  || (typeof data === 'object' && !Array.isArray(data)
      && Object.keys(data).length === 0);               // 空对象 {}
if (isEmpty) {
  return res.status(404).json({ code: 3001, message: '订单不存在' });
}
```

**为什么要分数组和对象判断：** `typeof []` 也是 `"object"`，所以要先排除数组，再判断对象。

---

### 修复 6：TC-GET-003 — 查询已取消订单拿到了 PENDING

**现象：** 查询 `{{order_id}}` 期望返回 status=CANCELLED，实际返回 status=PENDING。

**原因：** `{{order_id}}` 变量在测试过程中被反复覆写：
1. TC-ORDER-001 创建订单 → `order_id` = 20002（PENDING）
2. TC-GET-003 使用这个 order_id → 查到的是刚创建的 PENDING 订单

但测试目的是"查询一个已取消的订单"，所以需要固定的 CANCELLED 订单 ID。

**修复：** 把 URL 从 `{{base_url}}/orders/{{order_id}}` 改为 `{{base_url}}/orders/10003`（10003 是预置的 CANCELLED 订单）。

**教训：** 依赖可变状态的测试是不可靠的。测试要尽量减少互相之间的依赖（低耦合）。

---

### 修复 7：TC-GET-005 / TC-CANCEL-006 — 越权测试没有触发越权

**现象：** 用 testuser 的 token 去操作 orderId=10001，期望 403（越权拦截），实际返回 200。

**原因：** orderId=10001 的订单本来就属于 testuser（userId=1）。自己访问自己的订单，当然不会被拦截。

**修复：** 把目标改为 orderId=20001，这个订单属于 userId=2（用户 abcd）。用 testuser 的 token 去访问，就会触发越权检查。

**教训：** 测试"不被允许的事情"时，必须确保你用的测试数据真正触发了限制条件。

---

### 修复 8~10：TC-CANCEL-002/003/004 — 级联数据污染

**现象：** TC-CANCEL-001 把 `{{order_id}}` 的订单取消了。TC-CANCEL-002 再用同样的 `{{order_id}}` 去取消，返回 409（订单已被取消）。

**原因：** 这三条用例都用同一个变量 `{{order_id}}`。第一条取消了，后两条就没法取消了——就像三个人抢同一个苹果吃。

**修复：** 在每条取消用例的 pre-request 脚本中，先用 `pm.sendRequest` 创建一个全新的订单，把新订单的 ID 存到 `order_id`。这样每条用例都有自己的独立订单：
```javascript
pm.sendRequest({
    url: base_url + "/orders",
    method: "POST",
    header: { "Authorization": "Bearer " + token },
    body: { raw: JSON.stringify({ productId: 5001, quantity: 1, addressId: 100 }) }
}, function (err, res) {
    if (!err && res) {
        pm.environment.set("order_id", res.json().data.id);
    }
});
```

**为什么不用更简单的方案（比如给每个用例用不同的变量名）：**
因为 Postman Collection 的请求 URL 里写的是 `{{order_id}}`，如果每个用例用不同变量，就要修改很多 URL。不如在每个用例的 pre-request 里统一解决。

**`pm.sendRequest` 是什么：** Postman/Newman 提供的一个 API，可以在 pre-request 或 tests 脚本里发送额外的 HTTP 请求。它的回调机制是异步的——Newman 会等它完成后再发主请求。

```
目标：理解"在浏览器里输入网址后发生了什么"

学习内容：
```

## 七、核心术语速查表

| 术语 | 一句话解释 |
|------|-----------|
| HTTP | 浏览器和服务器之间说话的"语法规则" |
| API | 程序之间互相调用的"接口"（像插座——插上就能用） |
| REST | 一种 API 设计风格：用 HTTP 方法操作资源 |
| JSON | 一种数据格式，人和机器都能读懂 |
| Endpoint | API 的具体地址，如 POST /login |
| Request | 客户端发给服务器的消息 |
| Response | 服务器返回给客户端的消息 |
| Status Code | 三位数字，表示请求结果（200=成功，404=没找到，500=服务器崩了）|
| Header | 请求/响应的附加信息（如 token、内容类型）|
| Body | 请求/响应携带的实际数据 |
| Token | 登录后的"临时通行证" |
| Middleware | Express 中的"过滤器"，一层层处理请求 |
| npm | Node.js 的包管理器（应用商店） |
| Newman | Postman 的命令行版本 |
| Mock | "假的"——模拟真实服务的行为 |
| localhost | 本机地址（127.0.0.1） |
| Port | 端口号，区分同一台电脑上的不同服务 |
| CI/CD | 持续集成/持续部署——代码提交后自动测试和发布 |
| Assertion | 断言——"我认定结果应该是 X，不是就报错" |
| Pre-request | 发送请求前执行的脚本 |
| Environment | 一组变量，用于切换测试环境（dev/staging/prod）|
| Basic Auth | 一种简单的认证方式：用户名+密码用 Base64 编码后放请求头里 |
| Base64 | 一种编码方式，把二进制数据转成可打印字符（不是加密！）|
| Callback | 回调函数——"做完后通知我" |
| Async | 异步——不等待结果，先干别的，结果出来了再处理 |
