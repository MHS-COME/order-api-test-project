# OrderAPITest — 需求分析文档

> 版本：v2.0  
> 更新时间：2026-06-15  
> 角色：测试架构师

---

## 1. 项目概述

**OrderAPITest** 是一个全链路订单系统自动化测试平台，针对订单系统核心接口进行全面的功能验证、安全测试、UI 自动化测试与性能压测。测试范围覆盖用户注册、JWT 双 token 认证、订单生命周期管理（创建 → 查询 → 取消）、支付闭环（支付 → 查询 → 模拟）的完整链路。

### 1.1 测试目标

| 目标 | 说明 |
|------|------|
| 功能正确性 | 验证每个接口在合法输入下返回预期结果 |
| 异常容错性 | 验证参数缺失、未授权、资源不存在等异常场景的响应 |
| 数据一致性 | 验证订单状态变更在整个生命周期中的一致性 |
| 安全性 | 验证未授权访问、JWT 攻击、SQL 注入、XSS、越权访问被正确拦截 |
| UI 自动化 | 验证 Web 端注册/登录/下单/支付/取消完整用户旅程 |
| 性能基准 | 验证核心接口在并发负载下的响应时间和错误率 |

### 1.2 被测系统

- Mock 后端：Node.js + Express + jsonwebtoken + lowdb
- 前端：原生 HTML/CSS/JS SPA（订单管理仪表盘 v2.1）
- 协议：HTTP/HTTPS
- 数据格式：JSON
- 认证方式：JWT 双 token（access 1h + refresh 7d）
- 基础 URL（本地）：`http://localhost:3000`

### 2.1 用户注册 — `POST /register`

#### 请求规格

| 项目 | 内容 |
|------|------|
| URL | `/register` |
| 方法 | `POST` |
| Content-Type | `application/json` |
| 认证 | 无需 |

**请求体参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `username` | string | 是 | 用户名，4-32 字符 |
| `password` | string | 是 | 密码，6-20 字符 |
| `email` | string | 否 | 邮箱，需含 @ 和 . |

#### 预期成功响应

**HTTP Status: `200 OK`**

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 10007,
    "username": "newuser",
    "email": "new@test.com"
  }
}
```

#### 异常场景

| 场景编号 | 场景描述 | 预期 HTTP 状态 | 预期响应体（核心字段） |
|----------|---------|---------------|---------------------|
| REG-ERR-01 | 缺少必填字段 `username` | `400 Bad Request` | `{"code": 1001, "message": "参数校验失败：username 不能为空"}` |
| REG-ERR-02 | 缺少必填字段 `password` | `400 Bad Request` | `{"code": 1001, "message": "参数校验失败：password 不能为空"}` |
| REG-ERR-03 | username 长度 < 4 或 > 32 | `400 Bad Request` | `{"code": 1001, "message": "参数校验失败：username 长度须为 4-32 字符"}` |
| REG-ERR-04 | password 长度 < 6 或 > 20 | `400 Bad Request` | `{"code": 1001, "message": "参数校验失败：password 长度须为 6-20 字符"}` |
| REG-ERR-05 | 用户名已存在 | `409 Conflict` | `{"code": 1003, "message": "用户名已存在"}` |
| REG-ERR-06 | email 格式不正确 | `400 Bad Request` | `{"code": 1001, "message": "参数校验失败：email 格式不正确"}` |

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs...",
    "tokenType": "Bearer",
```

| `code` | int | 业务状态码，0 表示成功 |

| `data.refreshToken` | string | JWT 刷新令牌（refresh token，7d 有效） |
| `data.tokenType` | string | 令牌类型，固定为 "Bearer" |
| `data.expiresIn` | int | 访问令牌有效期（秒），默认 3600 |
| `data.userId` | int | 用户唯一标识 |
| `data.username` | string | 用户名 |

| 场景编号 | 场景描述 | 预期 HTTP 状态 | 预期响应体（核心字段） |
|----------|---------|---------------|---------------------|

| LOGIN-ERR-02 | 缺少必填字段 `password` | `400 Bad Request` | `{"code": 1001, "message": "参数校验失败：password 不能为空"}` |
| LOGIN-ERR-03 | 用户名或密码错误 | `401 Unauthorized` | `{"code": 1002, "message": "用户名或密码错误"}` |

## 3. 业务状态码汇总

| 业务码 | HTTP 状态 | 含义 | 所属模块 |
|--------|----------|------|---------|
| `0` | 200/201 | 操作成功 | 全部 |
| `1001` | 400 | 参数校验失败 | 全部（参数校验） |
| `1002` | 401 | 用户名或密码错误 | 登录 |
| `1003` | 409 | 用户名已存在 | 注册 |
| `1005` | 400 | 金额与订单实际金额不符 | 支付 |
| `2001` | 401 | 未提供认证令牌 / 无效令牌 | 认证中间件 |
| `2002` | 401 | 令牌已过期 | 认证中间件 |
| `2003` | 404 | 商品不存在或已下架 | 订单 |
| `2004` | 400 | 库存不足 | 订单 |
| `3001` | 404 | 订单不存在 | 订单/支付 |
| `3002` | 403 | 无权访问/操作该订单 | 订单/支付 |
| `4001` | 409 | 订单状态不允许取消（已发货） | 取消 |
| `4002` | 409 | 订单已被取消（重复操作） | 取消 |

## 4. 安全测试需求

### 4.1 认证安全

| 测试项 | 攻击向量 | 预期防护 |
|--------|---------|---------|
| 无 Token 访问 | 不携带 Authorization 头访问受保护接口 | 返回 401，code=2001 |
| 过期 Token | 使用已过期的 access token | 返回 401，code=2002 |
| 篡改 Token | 修改 JWT payload 后重签名 | 返回 401，code=2001 |
| none 算法攻击 | JWT header 中 alg 设为 none | 返回 401（服务端强制 HS256） |
| 签名缺失 | 移除 JWT 签名部分 | 返回 401，code=2001 |
| Refresh Token 撤销 | 使用被撤销的 refresh token（重新登录会使旧 refresh token 失效） | 返回 401，code=2001 |

### 4.2 注入攻击

| 测试项 | 攻击向量 | 预期防护 |
|--------|---------|---------|
| SQL 注入 | 参数中注入 `' OR '1'='1` | 返回 400（参数校验拦截）或 200（输出转义） |
| XSS 脚本注入 | remark 字段写入 `<script>alert(1)</script>` | HTML 实体转义输出 |
| 参数污染 | 同时传多个同名参数 | 取第一个值，正常处理 |

### 4.3 授权安全

| 测试项 | 攻击向量 | 预期防护 |
|--------|---------|---------|

## 6. 性能测试需求

### 6.1 压测策略

使用 k6 执行 3 阶段渐进式压测：

| 阶段 | 时长 | 虚拟用户（VU） | 说明 |
|------|------|---------------|------|
| 爬坡 | 30s | 0 → 20 | 逐步增加负载 |
| 保持 | 60s | 20 | 恒定负载，观察稳态表现 |
| 下降 | 30s | 20 → 0 | 逐步释放负载 |

### 6.2 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 总请求 P95 | ≤ 2000ms | 所有 HTTP 请求的 95 分位响应时间 |
| 登录 P95 | ≤ 1000ms | POST /login |
| 支付 P95 | ≤ 1000ms | POST /payment |
| 下单 P95 | ≤ 1500ms | POST /orders |
| 失败率 | < 1% | 所有请求的错误比例 |

### 6.3 自定义指标

| 指标名称 | 含义 |
|----------|------|
| `login_duration` | 登录接口耗时趋势 |
| `order_create_duration` | 创建订单接口耗时趋势 |
| `payment_duration` | 支付接口耗时趋势 |
| `e2e_flow_duration` | 端到端完整流程耗时趋势 |

## 7. 测试策略

### 7.1 测试分层

```
┌─────────────────────────────────────────────────┐
│           E2E 场景测试（全链路）                    │  ← 登录 → 创建 → 支付 → 查询 → 取消
├─────────────────────────────────────────────────┤
│           UI 自动化测试（Selenium）                 │  ← Web 端用户旅程 + 边界状态验证
├─────────────────────────────────────────────────┤
│           接口级功能测试（Postman/Newman）           │  ← 单接口正向 + 异常覆盖 + 安全测试
├─────────────────────────────────────────────────┤
│           性能测试（k6）                           │  ← 渐进式压测 + 自定义指标 + 阈值告警
└─────────────────────────────────────────────────┘
```

### 7.2 测试优先级

| 优先级 | 覆盖内容 |
|--------|---------|
| **P0** | 正向流程：注册 → 登录 → 创建订单 → 支付 → 查询 → 取消（全链路） |
| **P1** | 各接口必填参数缺失校验、未授权访问拦截、登录凭证校验 |
| **P2** | 越权访问、资源不存在、状态冲突、支付幂等性 |
| **P3** | 边界值（quantity 上下限）、JWT 攻击、XSS/SQL 注入、Token 过期 |

### 7.3 技术栈

| 组件 | 选型 | 用途 |
|------|------|------|
| Mock 后端 | Node.js + Express + jsonwebtoken + lowdb | JWT 双 token 认证 + 订单/支付 REST API |
| API 测试 | Postman + Newman + newman-reporter-html | 模块化接口测试，CSV 数据驱动 |
