## 3. 关键缺陷修复记录

本次迭代（v2.0 → v3.0）中识别并修复的关键缺陷：

### 前端缺陷

| 编号 | 缺陷 | 根因 | 修复方案 | 文件 | 状态 |
|------|------|------|---------|------|------|
| BUG-FE-001 | 订单列表取消订单后弹窗确认无效 | `showModal` 的 `onclick` 先调用 `closeModal()` 将 `modalCallback` 置为 null，之后判断失效 | 保存回调引用：`var cb = modalCallback; closeModal(); if (cb) cb();` | `frontend/index.html` | ✅ 已修复 |
| BUG-FE-002 | 登录成功后结果面板闪烁不消失 | 成功时 JS 立即跳转隐藏面板，`wait_result()` 检测面板可见性超时 | 改用 token 文本检测代替面板可见性：检查 token ≠ "未登录" 且长度 > 10 | `selenium_tests/test_ui.py` | ✅ 已修复 |
| BUG-FE-003 | 订单列表偶发加载为 0 行 | `load_order_list()` 等待"加载中"消失，但初始状态即无此文本，提前退出 | 改为等待 `<table>` 元素存在：`find_elements(By.CSS_SELECTOR, "table")` | `selenium_tests/test_ui.py` | ✅ 已修复 |

### 服务端缺陷

| 编号 | 缺陷 | 根因 | 修复方案 | 文件 | 状态 |
|------|------|------|---------|------|------|
| BUG-SRV-001 | Mock 空对象未返回 404 | `router.render` 只有 GET 方法才检查空数据 | 增加 `Object.keys(data).length === 0` 检测 | `mock-server/server.js` | ✅ 已修复 |
| BUG-SRV-002 | 取消用例级联数据污染 | 多个取消用例共用同一订单造成状态冲突 | 每个取消用例的 pre-request 中 `pm.sendRequest` 创建新订单 | Postman Collection | ✅ 已修复 |
| BUG-SRV-003 | remark 默认值不一致 | `remark \|\| ''` → 期望 null | 改为 `remark \|\| null` | `mock-server/server.js` | ✅ 已修复 |

## 4. 测试结论

### 结论：**通过 ✅**

> 本次全面回归测试在 Mock 环境下执行了 **73 个 API 请求、191 条断言，100% 通过**；**21 个 Selenium UI 自动化用例，100% 通过**；**k6 性能压测 3 阶段渐进式负载下所有核心指标远超目标阈值**。全链路 E2E 流程（注册→登录→创建→支付→查询→取消）正常，所有正向/反向/安全场景均验证正确。

### 验证覆盖的场景

- **用户注册**：合法注册、仅必填字段、缺少字段、用户名重复、边界值校验
- **用户认证**：合法登录、边界用户名/密码、缺少字段、错误凭证、Token 刷新（正常/无效/过期）
- **创建订单**：全字段/仅必填、quantity 1/999 边界、remark 200 边界、无 Token/过期/篡改、缺失字段/非法值
- **订单列表**：当前用户订单列表、按时间倒序、userId 过滤
- **查询订单**：本人订单/多商品/已取消、不存在 ID、越权、非数字 ID、无 Token
- **取消订单**：正常取消/附带原因/边界字符/持久化、不存在/越权、SHIPPED（409）、CANCELLED 幂等（409）、PAID 拒止（409）
- **支付模块**：正常支付、缺少字段、订单不存在、越权、重复支付幂等（409）、超时关闭（409）、金额校验（1005）、模拟支付成功/失败
- **安全测试**：SQL 注入、XSS 脚本注入、参数污染、无 Token/过期/篡改/none 算法/签名缺失、水平越权（查询/取消/支付）、Bearer 前缀移除
- **UI 自动化**：登录（正常/错误密码/空字段）、注册（成功/用户名过短）、创建订单（成功/未登录拦截）、订单列表（加载/状态标签/支付跳转/取消弹窗）、边界取消（PAID 拒止/CANCELLED 拒止）、支付（成功/空订单号）、全链路 E2E、侧边栏导航、仪表盘统计

## 5. 报告体系

| 报告类型 | 生成方式 | 输出路径 | 说明 |
|---------|---------|---------|------|
| Newman HTML | `newman run ... -r html` | `newman/report.html` | API 测试详细结果，含请求/响应/断言 |
| Newman JSON | `newman run ... -r json` | `newman/results.json` | 机器可读格式，用于 Allure 转换 |
| Allure HTML | `allure generate` | `allure-report/` | 专业可视化报告，含趋势图、严重性分类 |
| Selenium 控制台 | `pytest -v` | 标准输出 | 21 个 UI 用例的实时执行日志 |
| k6 终端 | `k6 run` | 标准输出 | 性能指标汇总 + 阈值检查结果 |

### 生成 Allure 报告命令

```bash
# 1. 先跑 Newman 生成 JSON 报告
newman run postman/order_api_collection.json \
  -e postman/environments/dev.environment.json \
  -r json --reporter-json-export newman/results.json

# 2. 转换为 Allure 结果
node newman-to-allure.js --report newman/results.json

# 3. 生成 Allure HTML 报告
allure generate allure-results -o allure-report --clean

# 4. 打开报告
allure open allure-report
```
