## 1. 测试范围

### 1.1 功能测试

| 模块 | 接口 | 正向用例 | 异常用例 | 说明 |
|------|------|---------|---------|------|
| 用户认证 | `POST /login` | 合法凭证登录成功 | 4 | 覆盖参数缺失、凭证错误、账户锁定 |
| 订单创建 | `POST /orders` | 创建订单成功 | 7 | 覆盖认证、参数校验、商品/库存异常 |
| 订单查询 | `GET /orders/{id}` | 查询订单详情成功 | 4 | 覆盖越权、资源不存在、格式校验 |
| 订单取消 | `PUT /orders/{id}/cancel` | 取消订单成功 | 5 | 覆盖状态机冲突、幂等性、越权 |
| **全链路场景** | 串联上述 4 个接口 | 1 条 E2E 链路 | — | 登录 → 创建 → 查询 → 取消 |

**总计：5 条正向用例 + 20 条异常用例 = 25 条接口用例**

#### 功能测试通过标准

- 所有 P0/P1 用例通过率 **100%**
- 所有 P2 用例通过率 **≥ 98%**
- 所有 P3 用例通过率 **≥ 95%**
- 无 Blocker/Critical 级别缺陷未关闭

### 1.2 性能测试（基础要求）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单接口响应时间（P95） | ≤ 500ms | `GET /orders/{id}` 等读接口 |
| 单接口响应时间（P95） | ≤ 800ms | `POST /orders` 等写接口 |
| 并发用户数 | 50 | 模拟峰值业务场景 |
| 错误率 | < 0.1% | 在目标并发下 |
| TPS（每秒事务数） | ≥ 100 | `/login` 接口 |

> 注：性能测试在接口自动化之后执行，采用轻量级压测（非全链路压测），重点发现慢查询和接口超时瓶颈。

### 1.3 安全测试（基础要求）

| 检查项 | 验证方式 | 期望结果 |
|--------|---------|---------|
| 未授权访问 | 不带 Token 调用需认证的接口 | 返回 401 |
| Token 伪造/篡改 | 使用无效 JWT 调用接口 | 返回 401 |
| 越权访问 | 用户 A 的 Token 访问用户 B 的订单 | 返回 403 |
| SQL 注入 | 在参数中注入 `' OR '1'='1` | 返回 400（被参数校验拦截） |
| XSS | 在 `remark` 字段写入 `<script>alert(1)</script>` | 返回 200，响应体中脚本被转义 |
| 敏感信息泄露 | 检查响应体中是否含密码、Token 原文等 | 不包含 |

## 2. 测试策略

### 2.1 接口测试优先级矩阵

```
            影响高
              │
    P1 ★★★    │    P0 ★★★★
   (重要但不紧急) │   (紧急且重要)
              │
  ────────────┼──────────── 影响低
              │
    P3 ★      │    P2 ★★
   (低优先级)  │   (一般)
              │
           紧急低
```

| 优先级 | 定义 | 覆盖内容 | 占比 |
|--------|------|---------|------|
| **P0** | 核心业务流程，阻塞即不可发布 | 登录 → 创建订单 → 查询 → 取消 全链路 | 20% |
| **P1** | 高频接口的关键异常路径 | 必填参数缺失、未授权访问、凭证错误 | 40% |
| **P2** | 一般异常场景 | 越权访问、资源不存在、状态冲突 | 28% |
| **P3** | 边界值与罕见场景 | 参数边界值、幂等性、Token 过期、账户锁定 | 12% |

### 2.2 自动化覆盖目标

```
                    手动测试 (探索性 + 新功能)
                   ┌──────────────────┐
                   │     10%          │
                   │                  │
      ┌────────────┼──────────────────┼────────────┐
      │            │                  │            │
      │  自动化 90% │    P0 自动化    │  自动化 70% │
      │            │    目标 100%     │            │
      │            │                  │            │
      └────────────┼──────────────────┼────────────┘
                   │                  │
                   │    P3 自动化     │
                   │    目标 50%      │
                   └──────────────────┘
```

| 优先级 | 自动化目标 | 说明 |
|--------|-----------|------|
| P0 | **100%** | 每次提交必须通过，作为 CI 门禁 |
| P1 | **90%** | 每日构建自动执行 |
| P2 | **70%** | 每轮回归自动执行 |
| P3 | **50%** | 发版前手动触发 |


## 4. 风险与应对

| 编号 | 风险项 | 概率 | 影响 | 等级 | 应对措施 |
|------|--------|------|------|------|---------|
| R1 | **测试环境不稳定**（服务宕机、网络中断导致用例大量误报） | 中 | 高 | 🔴 高 | ① 执行前自动健康检查（ping 基础 URL + `/health`）<br>② 失败用例自动重试 1 次<br>③ 环境不可用时跳过后续执行并通知 |
| R2 | **被测接口变更未同步**（字段改名、路径调整导致用例大量失败） | 高 | 中 | 🟡 中 | ① 引入 Schema 契约测试，变更时先更新 Collection<br>② 要求开发在 PR 中标注 breaking change<br>③ 用例与 API 文档使用同一数据源（如 OpenAPI Spec） |
| R3 | **测试数据污染**（并发执行时数据相互干扰） | 中 | 中 | 🟡 中 | ① 每次执行前初始化独立测试数据集<br>② 使用 `timestamp` + `random` 后缀生成唯一数据<br>③ 执行后自动清理（在 Collection 的 `pre-request` / `test` 脚本中处理） |
| R4 | **第三方依赖不可用**（短信、支付网关等 Mock 不稳定） | 低 | 中 | 🟡 中 | ① 核心订单流程不依赖真实第三方<br>② 第三方依赖使用 Mock Server（如 WireMock）<br>③ Mock 故障时跳过该用例并标记为 `skipped`（不计入失败率） |
| R5 | **Token 过期导致的测试中断**（长链路执行超过 Token 有效期） | 中 | 低 | 🟢 低 | ① 在每条用例的 `pre-request` 脚本中检查 Token 是否临近过期<br>② 过期前自动重新调用 `/login` 刷新 Token<br>③ 设置 Token 有效期为 2 小时（和开发协商） |
| R6 | **Jenkins 执行机资源不足**（磁盘满、内存溢出） | 低 | 高 | 🟡 中 | ① 设置日志保留策略（最近 30 次构建）<br>② Newman 执行时禁用 `--verbose`，减少日志体积<br>③ 配置磁盘告警阈值（> 85% 通知） |
